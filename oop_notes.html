01

Introduction to OOP

Object-Oriented Programming is a programming paradigm built around objects rather than just functions and logic. It mirrors how we think about the real world.

  • An object represents any real-world entity — a pen, a laptop, a car, a bank account.
  • OOP is not mandatory, but makes code more organized, reusable, and maintainable — valuable at company scale.
  • Core STL/standard-library types (vector, string, Python's list, dict) are internally built using OOP.
  • OOP concepts are language-agnostic — the same ideas apply in C++, Java, Python; only syntax changes.
PillarOne-line meaning
EncapsulationBundling data + methods, restricting direct access to data
AbstractionHiding implementation details, showing only essential features
InheritanceAcquiring properties/behavior of one class into another
PolymorphismOne entity behaving in multiple forms
Interview note: almost every placement interview asks OOP fundamentals — definitions, differences, and small code-tracing questions.
02

Class & Object

A class is a blueprint for creating objects — it defines properties (data) and behavior (methods). An object is an instance of a class, occupying real memory.

Without classes, storing data for 100 teachers would need 100 sets of separate variables. A class defines the structure once; you create as many objects as needed.

#include <iostream>
#include <string>
using namespace std;

class Teacher {
    public:
        string name;
        string department;
        int salary;

        void displayInfo() {
            cout << "Name: " << name << endl;
            cout << "Department: " << department << endl;
            cout << "Salary: " << salary << endl;
        }
};

int main() {
    Teacher t1;
    t1.name = "Mr. Sharma";
    t1.department = "Computer Science";
    t1.salary = 60000;

    Teacher t2;
    t2.name = "Ms. Verma";
    t2.department = "Mathematics";
    t2.salary = 55000;

    t1.displayInfo();
    t2.displayInfo();
    return 0;
}
class Teacher:
    def __init__(self, name, department, salary):
        self.name = name
        self.department = department
        self.salary = salary

    def display_info(self):
        print(f"Name: {self.name}")
        print(f"Department: {self.department}")
        print(f"Salary: {self.salary}")


t1 = Teacher("Mr. Sharma", "Computer Science", 60000)
t2 = Teacher("Ms. Verma", "Mathematics", 55000)

t1.display_info()
t2.display_info()
Key idea: Teacher is the blueprint. t1/t2 are objects — each with its own data, sharing the same methods. Python has no separate constructor keyword; __init__ plays that role.

Access Specifiers

Access specifiers control visibility of class members.

SpecifierWithin classDerived classOutside class
public✓✓✓
protected✓✓✗
private✓✗✗

Default for class is private; default for struct is public.

class Account {
    private:
        double balance;      // hidden

    protected:
        int accountNumber;   // accessible in derived classes

    public:
        string ownerName;    // accessible everywhere

        void setBalance(double b) { balance = b; }
        double getBalance() { return balance; }
};

int main() {
    Account a;
    a.ownerName = "Rahul";     // OK - public
    a.setBalance(5000);        // OK - public method
    // a.balance = 5000;       // ERROR - private
    return 0;
}
class Account:
    def __init__(self, owner_name):
        self.owner_name = owner_name        # public (convention: no prefix)
        self._account_number = None          # protected (convention: single underscore)
        self.__balance = 0.0                 # private (name-mangled: _Account__balance)

    def set_balance(self, b):
        self.__balance = b

    def get_balance(self):
        return self.__balance


a = Account("Rahul")
a.owner_name = "Rahul"      # OK - public
a.set_balance(5000)          # OK - public method
# a.__balance = 5000        # Does NOT raise, but creates a NEW attribute (no true privacy)
Python has no true access enforcement. _name (single underscore) is a convention for "protected," and __name (double underscore) triggers name-mangling to _ClassName__name, but nothing is truly private — it can still be accessed if you know the mangled name.
04

Encapsulation

Encapsulation = wrapping data and the functions that operate on it into a single unit (class), restricting direct access to some components.

  • Achieved via access specifiers + public getter/setter methods.
  • Protects data integrity — state can only change through controlled methods.
  • Analogy: a capsule medicine — ingredients (data) are hidden inside; you interact with the capsule as a whole.
class BankAccount {
    private:
        double balance;

    public:
        BankAccount(double initialBalance) { balance = initialBalance; }

        void deposit(double amount) {
            if (amount > 0) balance += amount;
        }

        void withdraw(double amount) {
            if (amount > 0 && amount <= balance) balance -= amount;
            else cout << "Invalid withdrawal amount!" << endl;
        }

        double getBalance() { return balance; }
};

int main() {
    BankAccount acc(1000);
    acc.deposit(500);
    acc.withdraw(200);
    cout << "Balance: " << acc.getBalance() << endl;
    // acc.balance = -99999;  // ERROR - private
    return 0;
}
OutputBalance: 1300
class BankAccount:
    def __init__(self, initial_balance):
        self.__balance = initial_balance   # private

    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount

    def withdraw(self, amount):
        if 0 < amount <= self.__balance:
            self.__balance -= amount
        else:
            print("Invalid withdrawal amount!")

    def get_balance(self):
        return self.__balance


acc = BankAccount(1000)
acc.deposit(500)
acc.withdraw(200)
print(f"Balance: {acc.get_balance()}")
# Direct acc.__balance = -99999 would NOT touch the real balance
# (it creates a new unrelated attribute due to name mangling)
OutputBalance: 1300
Without encapsulation, external code could directly corrupt state (e.g. set a negative balance). Encapsulation forces all changes through validated methods.
05

Constructors

A constructor is a special member function automatically called when an object is created, used to initialize data members.

  • Same name as the class (C++); no return type at all.
  • Automatically invoked at object creation.
  • Can be overloaded in C++ (multiple constructors, different parameters).
TypeDescription
Default ConstructorTakes no parameters; compiler provides one if none defined
Parameterized ConstructorTakes arguments to initialize members with specific values
Copy ConstructorInitializes an object using another object of the same class
class Student {
    public:
        string name;
        int age;

        // Default constructor
        Student() {
            name = "Unknown";
            age = 0;
            cout << "Default constructor called" << endl;
        }

        // Parameterized constructor
        Student(string n, int a) {
            name = n;
            age = a;
            cout << "Parameterized constructor called" << endl;
        }
};

int main() {
    Student s1;                  // default constructor
    Student s2("Aman", 20);      // parameterized constructor

    cout << s1.name << " " << s1.age << endl;
    cout << s2.name << " " << s2.age << endl;
    return 0;
}
class Student:
    # Python has ONE __init__, but default args simulate overloading
    def __init__(self, name="Unknown", age=0):
        self.name = name
        self.age = age
        print("Constructor called")


s1 = Student()               # uses defaults -> "Unknown", 0
s2 = Student("Aman", 20)      # parameterized

print(s1.name, s1.age)
print(s2.name, s2.age)
Python cannot overload __init__ the way C++ overloads constructors — a class has exactly one __init__. Default parameter values, *args/**kwargs, or classmethod "factory" functions are used instead to simulate multiple constructors.
06

The this Pointer

this is an implicit pointer inside every non-static C++ member function, pointing to the current object. Used to:

  1. Resolve naming conflicts between member variables and parameters.
  2. Return the current object (method chaining).
  3. Pass the current object’s address elsewhere.
class Box {
    public:
        int length;

        Box(int length) {
            this->length = length;   // this->length = member, length = param
        }

        void show() {
            cout << "Length: " << this->length << endl;
        }

        Box& setLength(int length) {
            this->length = length;
            return *this;             // enables chaining
        }
};

int main() {
    Box b(10);
    b.show();                 // Length: 10
    b.setLength(20).show();   // Length: 20
    return 0;
}
class Box:
    def __init__(self, length):
        self.length = length     # 'self' is Python's explicit equivalent of 'this'

    def show(self):
        print(f"Length: {self.length}")

    def set_length(self, length):
        self.length = length
        return self               # enables chaining, like returning *this

b = Box(10)
b.show()                          # Length: 10
b.set_length(20).show()           # Length: 20
Python equivalent: self is explicit and must be the first parameter of every instance method — Python never hides it the way C++ hides this. There is no dereferencing needed (self is already the object, not a pointer).
07

Copy Constructor

A copy constructor creates a new object as a copy of an existing object of the same class.

Called when: (1) an object is initialized from another object, (2) an object is passed by value, (3) an object is returned by value.

class Student {
    public:
        string name;
        int age;

        Student(string n, int a) { name = n; age = a; }

        // User-defined copy constructor
        Student(const Student &s) {
            name = s.name;
            age = s.age;
            cout << "Copy constructor called" << endl;
        }
};

int main() {
    Student s1("Priya", 21);
    Student s2 = s1;    // Copy constructor invoked

    cout << s2.name << " " << s2.age << endl;
    return 0;
}
import copy

class Student:
    def __init__(self, name, age):
        self.name = name
        self.age = age


s1 = Student("Priya", 21)

# Plain assignment does NOT copy - it just aliases the same object!
s2_alias = s1
s2_alias.name = "Changed"
print(s1.name)   # "Changed" too! s1 and s2_alias are the SAME object

# To actually copy, use the copy module
s3 = copy.copy(s1)     # shallow copy - new object, own top-level attributes
print(s3.name, s3.age)  # "Changed" 21 - independent from s1 now
Big difference: in C++, Student s2 = s1; always creates a real copy via the copy constructor. In Python, plain assignment (s2 = s1) never copies — it just binds a second name to the same object. Real copies need the copy module: copy.copy() (shallow) or copy.deepcopy() (deep).
08

Shallow Copy vs Deep Copy

One of the most frequently asked OOP interview topics, especially for classes managing dynamic memory / references.

AspectShallow CopyDeep Copy
What is copiedPointer / reference addressActual data pointed to
MemoryShared between objectsIndependent per object
RiskDangling pointer, double-freeNone (if done correctly)
class Deep {
    public:
        int* data;

        Deep(int val) { data = new int(val); }

        // User-defined DEEP copy constructor
        Deep(const Deep &source) {
            data = new int(*source.data);  // new memory, copy the VALUE
        }

        ~Deep() { delete data; }   // safe - each object owns its own memory
};

int main() {
    Deep d1(100);
    Deep d2 = d1;          // deep copy
    *d2.data = 200;

    cout << "d1.data = " << *d1.data << endl;  // 100 (unaffected)
    cout << "d2.data = " << *d2.data << endl;  // 200
    return 0;
}
Outputd1.data = 100\nd2.data = 200
import copy

class Deep:
    def __init__(self, val):
        self.data = [val]     # a mutable container, like a "pointer" to a value

d1 = Deep(100)

d2_shallow = copy.copy(d1)
d2_shallow.data[0] = 999
print(d1.data)          # [999] - CHANGED! shallow copy shared the inner list

d1 = Deep(100)  # reset
d2_deep = copy.deepcopy(d1)
d2_deep.data[0] = 200
print(d1.data)           # [100] - unaffected
print(d2_deep.data)      # [200] - independent
Output[999] [100] [200]
Python parallel: copy.copy() mirrors C++’s default (shallow, compiler-generated) copy constructor — nested mutable objects (lists, dicts, custom objects) are shared. copy.deepcopy() mirrors a hand-written C++ deep copy constructor — it recursively copies everything.
09

Destructor

A destructor is automatically called when an object goes out of scope or is deleted, used to release resources.

  • Same name as class, prefixed with ~ in C++; no parameters, no return type.
  • Cannot be overloaded — exactly one destructor per class.
  • Called in reverse order of creation for stack objects going out of scope.
class Resource {
    public:
        int* ptr;

        Resource(int val) {
            ptr = new int(val);
            cout << "Constructor: resource acquired" << endl;
        }

        ~Resource() {
            delete ptr;
            cout << "Destructor: resource released" << endl;
        }
};

int main() {
    {
        Resource r(50);
        cout << "Inside scope, value = " << *r.ptr << endl;
    } // destructor auto-called here

    cout << "Outside scope" << endl;
    return 0;
}
OutputConstructor: resource acquired\nInside scope, value = 50\nDestructor: resource released\nOutside scope
class Resource:
    def __init__(self, val):
        self.val = val
        print("Constructor: resource acquired")

    def __del__(self):
        print("Destructor: resource released")


def demo():
    r = Resource(50)
    print(f"Inside scope, value = {r.val}")
    # __del__ called when r's reference count hits 0
    # (typically when function ends / object is garbage collected)

demo()
print("Outside scope")
OutputConstructor: resource acquired Inside scope, value = 50 Destructor: resource released Outside scope
Important difference: C++ destructors run deterministically the instant an object goes out of scope (stack unwinding). Python's __del__ runs when the garbage collector reclaims the object — timing is not guaranteed, especially with reference cycles. For deterministic cleanup in Python, use with statements / context managers instead of relying on __del__.

Inheritance

Inheritance lets a derived/child class acquire the properties and behaviors of a base/parent class — models real-world "is-a" relationships and promotes code reuse.

class Vehicle {
    public:
        string brand;
        void honk() { cout << brand << " vehicle honks!" << endl; }
};

class Car : public Vehicle {
    public:
        int numDoors;
        void showDoors() {
            cout << "This car has " << numDoors << " doors." << endl;
        }
};

int main() {
    Car c;
    c.brand = "Toyota";     // inherited
    c.numDoors = 4;

    c.honk();        // Toyota vehicle honks!
    c.showDoors();   // This car has 4 doors.
    return 0;
}
class Vehicle:
    def __init__(self, brand):
        self.brand = brand

    def honk(self):
        print(f"{self.brand} vehicle honks!")


class Car(Vehicle):              # Car inherits from Vehicle
    def __init__(self, brand, num_doors):
        super().__init__(brand)   # call the parent constructor
        self.num_doors = num_doors

    def show_doors(self):
        print(f"This car has {self.num_doors} doors.")


c = Car("Toyota", 4)
c.honk()          # Toyota vehicle honks!  (inherited method)
c.show_doors()    # This car has 4 doors.
class Car : public Vehicle in C++ corresponds to class Car(Vehicle): in Python. C++’s implicit base-constructor call is explicit in Python via super().__init__(...).

Modes of Inheritance

The access specifier used while inheriting determines how base class members are accessible in the derived class.

Base memberpublic inheritanceprotected inheritanceprivate inheritance
publicpublicprotectedprivate
protectedprotectedprotectedprivate
privateinaccessibleinaccessibleinaccessible
class Base {
    public:    int a;
    protected: int b;
    private:   int c;
};

class PublicDerived    : public    Base { /* a->public, b->protected, c->inaccessible */ };
class ProtectedDerived : protected Base { /* a->protected, b->protected, c->inaccessible */ };
class PrivateDerived   : private   Base { /* a->private, b->private, c->inaccessible */ };
# Python has no "inheritance mode" keyword (public/protected/private inheritance).
# Inheritance is always effectively "public" - the derived class gets everything
# except name-mangled __private attributes are still access-restricted by mangling.

class Base:
    def __init__(self):
        self.a = 1          # public-by-convention
        self._b = 2         # protected-by-convention
        self.__c = 3        # private-by-convention (name-mangled)

class Derived(Base):
    def show(self):
        print(self.a)        # OK
        print(self._b)        # OK (convention allows it)
        # print(self.__c)     # AttributeError - mangled to _Base__c
        print(self._Base__c)  # works, but you're not "supposed to" do this
Python has no equivalent of public/protected/private inheritance modes — there is only one style of inheritance. Encapsulation strictness in Python comes entirely from naming convention, not from the inheritance declaration itself.

Types of Inheritance

TypeDescription
SingleOne derived class inherits from one base class
MultipleOne derived class inherits from more than one base class
MultilevelA class derived from a class, itself derived from another (chain)
HierarchicalMultiple derived classes inherit from a single base class
HybridCombination of two or more types above
// Multiple Inheritance
class Flyable {
    public: void fly() { cout << "can fly" << endl; }
};
class Swimmable {
    public: void swim() { cout << "can swim" << endl; }
};
class Duck : public Flyable, public Swimmable {
    // Duck can both fly() and swim()
};

// Multilevel Inheritance
class LivingBeing { public: void breathe() { cout << "breathes" << endl; } };
class Animal : public LivingBeing { public: void eat() { cout << "eats" << endl; } };
class Dog : public Animal { public: void bark() { cout << "barks" << endl; } };
// Dog has breathe(), eat(), AND bark()

// Hybrid (also causes the "diamond problem")
class A { public: void a() {} };
class B : public A { public: void b() {} };
class C : public A { public: void c() {} };
class D : public B, public C { public: void d() {} };
// Fix with virtual inheritance:
// class B : virtual public A {};
// class C : virtual public A {};
# Multiple Inheritance - Python supports this natively and simply
class Flyable:
    def fly(self): print("can fly")

class Swimmable:
    def swim(self): print("can swim")

class Duck(Flyable, Swimmable):     # multiple inheritance - no extra syntax needed
    pass

d = Duck()
d.fly()    # can fly
d.swim()   # can swim

# Multilevel Inheritance
class LivingBeing:
    def breathe(self): print("breathes")

class Animal(LivingBeing):
    def eat(self): print("eats")

class Dog(Animal):
    def bark(self): print("barks")

dog = Dog()
dog.breathe(); dog.eat(); dog.bark()

# "Diamond problem" - Python solves this automatically via
# the C3 Linearization algorithm (Method Resolution Order / MRO)
class A:
    def hello(self): print("A")

class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.mro())  # shows the exact resolution order Python computes automatically
Diamond problem: C++ requires manual virtual inheritance to avoid duplicate base-class copies. Python resolves this automatically using the MRO (Method Resolution Order) via C3 linearization — call ClassName.mro() or ClassName.__mro__ to inspect it.
13

Polymorphism

Polymorphism = "many forms" — the same interface behaves differently depending on context.

TypeAlso calledResolved at
Compile-time PolymorphismStatic / Early bindingCompile time
Runtime PolymorphismDynamic / Late bindingRun time

Compile-time: Function Overloading, Operator Overloading. Runtime: Function Overriding + Virtual Functions.

Python is dynamically typed, so it doesn’t really have "compile-time" polymorphism the way C++ does. Python achieves flexible behavior mainly via duck typing (see below) and method overriding — both are effectively resolved at runtime.
14

Function Overloading

Function Overloading: multiple functions with the same name, different parameter lists, in the same scope. Compile-time polymorphism.

  • Functions must differ in number and/or type of parameters.
  • Return type alone cannot distinguish overloaded functions.
class Calculator {
    public:
        int add(int a, int b) { return a + b; }
        double add(double a, double b) { return a + b; }
        int add(int a, int b, int c) { return a + b + c; }
};

int main() {
    Calculator calc;
    cout << calc.add(2, 3) << endl;         // add(int,int) -> 5
    cout << calc.add(2.5, 3.5) << endl;     // add(double,double) -> 6.0
    cout << calc.add(1, 2, 3) << endl;      // add(int,int,int) -> 6
    return 0;
}
Output5\n6\n6
# Python does NOT support true function overloading -
# a later definition of the same name simply overwrites the earlier one.
# Workarounds: default arguments, *args, or functools.singledispatch

class Calculator:
    def add(self, a, b, c=None):
        if c is not None:
            return a + b + c
        return a + b


calc = Calculator()
print(calc.add(2, 3))        # 5
print(calc.add(2.5, 3.5))    # 6.0
print(calc.add(1, 2, 3))     # 6

# More "proper" overloading using functools.singledispatch (module-level functions)
from functools import singledispatch

@singledispatch
def describe(arg):
    print(f"Unknown type: {arg}")

@describe.register
def _(arg: int):
    print(f"Integer: {arg}")

@describe.register
def _(arg: str):
    print(f"String: {arg}")

describe(5)      # Integer: 5
describe("hi")   # String: hi
Output5 6.0 6 Integer: 5 String: hi
Key interview point: Python has no native function overloading — defining add() twice just replaces the first definition. Simulate it with default/variadic arguments, or use functools.singledispatch for real type-based dispatch.
15

Function Overriding

Function Overriding: a derived class redefines a base class member function with the same name, parameters, and return type. Core to runtime polymorphism.

class Animal {
    public:
        void sound() { cout << "Animal makes a sound" << endl; }
};

class Dog : public Animal {
    public:
        void sound() { cout << "Dog barks" << endl; }   // overrides
};

int main() {
    Dog d;
    d.sound();          // Dog barks

    Animal* aptr = &d;
    aptr->sound();       // WITHOUT virtual: calls Animal::sound() - wrong!
    return 0;
}
OutputDog barks\nAnimal makes a sound
class Animal:
    def sound(self):
        print("Animal makes a sound")

class Dog(Animal):
    def sound(self):          # overrides Animal.sound()
        print("Dog barks")

d = Dog()
d.sound()          # Dog barks

a_ref: Animal = d
a_ref.sound()      # Dog barks - ALWAYS correct in Python, no "virtual" needed!
OutputDog barks Dog barks
Major difference: in C++, overriding through a base-class pointer only works correctly if the base method is virtual (see next topic) — otherwise it silently calls the base version (static binding). In Python, all method calls are dynamically dispatched by default — there is no separate "virtual" keyword needed; overriding always works correctly.
16

Virtual Functions

A virtual function tells the C++ compiler to support runtime polymorphism — the function called is determined by the actual object type, resolved at runtime via a vtable.

class Animal {
    public:
        virtual void sound() { cout << "Animal makes a sound" << endl; }
        virtual ~Animal() {}         // virtual destructor - important!
};

class Dog : public Animal {
    public:
        void sound() override { cout << "Dog barks" << endl; }
};

class Cat : public Animal {
    public:
        void sound() override { cout << "Cat meows" << endl; }
};

int main() {
    Animal* animals[3];
    Dog d; Cat c; Animal a;
    animals[0] = &a; animals[1] = &d; animals[2] = &c;

    for (int i = 0; i < 3; i++) {
        animals[i]->sound();   // correct derived version called - runtime polymorphism!
    }
    return 0;
}
OutputAnimal makes a sound\nDog barks\nCat meows
class Animal:
    def sound(self):
        print("Animal makes a sound")

class Dog(Animal):
    def sound(self):
        print("Dog barks")

class Cat(Animal):
    def sound(self):
        print("Cat meows")


animals = [Animal(), Dog(), Cat()]
for a in animals:
    a.sound()   # correct version always called - Python has NO 'virtual' keyword
                # because every method lookup already happens dynamically at runtime
OutputAnimal makes a sound Dog barks Cat meows
Python has no virtual keyword because every method call is looked up dynamically on the actual object’s type at runtime (this is just how Python’s attribute lookup works). What C++ has to opt into with virtual, Python does unconditionally.
17

Abstraction

Abstraction = hiding internal implementation details, showing only essential features. Focuses on what an object does, not how.

class Car {
    private:
        void startEngine() {       // hidden internal detail
            cout << "Engine internals starting..." << endl;
        }

    public:
        void drive() {              // exposed simplified interface
            startEngine();
            cout << "Car is now driving" << endl;
        }
};

int main() {
    Car c;
    c.drive();          // user only interacts with drive()
    // c.startEngine(); // ERROR - hidden
    return 0;
}
OutputEngine internals starting...\nCar is now driving
class Car:
    def __start_engine(self):        # name-mangled "private" - hidden internal detail
        print("Engine internals starting...")

    def drive(self):                  # exposed simplified interface
        self.__start_engine()
        print("Car is now driving")


c = Car()
c.drive()
# c.__start_engine()   # AttributeError (name-mangled to _Car__start_engine)
OutputEngine internals starting... Car is now driving
Duck typing note: Python leans heavily on duck typing for abstraction — "if it walks like a duck and quacks like a duck, treat it as a duck." Code often doesn’t check an object’s exact type at all, just that it has the method being called.
18

Abstract Class

An abstract class cannot be instantiated directly — used only as a base class. It contains at least one pure virtual function in C++ (= 0), which derived classes must override.

class Shape {   // abstract class
    public:
        virtual double area() = 0;      // pure virtual - no body
        virtual void display() {         // regular virtual - has default body
            cout << "This is a shape" << endl;
        }
};

class Circle : public Shape {
    private:

    public:

};

        Rectangle(double l, double w) : length(l), width(w) {}

from abc import ABC, abstractmethod

class Shape(ABC):                     # inherit from ABC to make it abstract
    @abstractmethod
    def area(self):                    # pure virtual equivalent - no implementation
        pass

    def display(self):                 # regular method - has a default body
        print("This is a shape")


class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return 3.14159 * self.radius ** 2


class Rectangle(Shape):
    def __init__(self, length, width):
        self.length = length
        self.width = width

    def area(self):
        return self.length * self.width

Static Keyword

static changes the lifetime/scope of variables or functions.

  • Static data members: shared by all objects of a class — one copy total.
  • Static member functions: callable without an object; can only access static data.
  • Static local variables: retain value between function calls.
class Counter {
    private:
        static int count;      // declaration - shared across all objects

    public:
        Counter() { count++; }
        static int getCount() { return count; }   // static member function
};

int Counter::count = 0;   // definition, required outside the class

int main() {
    Counter c1, c2, c3;
    cout << "Total objects created: " << Counter::getCount() << endl;  // 3
    return 0;
}
OutputTotal objects created: 3
class Counter:
    count = 0    # class variable - shared across ALL instances (like C++ static member)

    def __init__(self):
        Counter.count += 1     # modify via the class, not self

    @staticmethod
    def get_count():            # static method - no 'self', callable without an instance
        return Counter.count


c1 = Counter()
c2 = Counter()
c3 = Counter()

print(f"Total objects created: {Counter.get_count()}")   # 3
# Also callable as: Counter.get_count()
OutputTotal objects created: 3
Mapping: C++ static int count; (member) ↔ Python class variable (count = 0 defined directly in the class body, shared by all instances). C++ static member function ↔ Python @staticmethod (or @classmethod if it needs access to the class itself via cls).
