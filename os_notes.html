Introduction to OS

An Operating System (OS) is system software that acts as an intermediary between computer hardware and the user, managing hardware resources and providing services for application programs.

Core Functions

  • Process Management — creating, scheduling, and terminating processes.
  • Memory Management — allocating and deallocating memory space as needed.
  • File System Management — organizing and controlling access to files/directories.
  • Device Management — controlling I/O devices via drivers.
  • Security & Access Control — protecting data and resources from unauthorized access.

Types of Operating Systems

TypeDescriptionExample use case
Batch OSJobs with similar needs are grouped and run as a batch, no direct user interaction during executionPayroll processing, old mainframes
Multiprogramming OSMultiple programs reside in memory at once; CPU switches between them to maximize utilizationEarly time-shared mainframes
Time-Sharing OSCPU time is divided into small slices (time quantum) shared among multiple users/processes, giving illusion of simultaneous executionMulti-user Unix systems
Multiprocessing OSTwo or more CPUs work together, sharing memory bus and peripherals, for increased throughputModern multi-core systems
Real-Time OS (RTOS)Guarantees processing within a strict, predictable time constraint (hard or soft real-time)Pacemakers, industrial control systems
Distributed OSManages a group of independent computers, making them appear as a single system to usersGoogle's distributed backends

Key Distinction: Multiprogramming vs Multitasking

Multiprogramming focuses on maximizing CPU utilization by keeping multiple jobs in memory so the CPU always has something to execute (especially during I/O waits) — but from the user’s perspective it isn’t necessarily interactive. Multitasking is multiprogramming extended with rapid context switching (time-sharing) so users perceive several programs running simultaneously.

Interview tip: A very common question is "OS as a resource manager vs OS as an extended machine." As a resource manager, the OS allocates CPU, memory, and I/O fairly and efficiently among competing processes. As an extended/virtual machine, the OS hides messy hardware details and gives programmers a clean, convenient abstraction (e.g., read()/write() instead of raw disk-controller instructions).

Q1.What is the primary role of an Operating System?
The OS's core job is resource management and abstraction — it sits between raw hardware and the programs/users that need to use that hardware, handling CPU, memory, I/O, and files on their behalf.
Q2.In a Batch Operating System, how are jobs typically processed?
Batch OS groups jobs with similar requirements and runs them sequentially without user intervention during execution — this was common on early mainframes for things like payroll runs.
Q3.Which OS type guarantees that a task will be completed within a strict, predictable time frame?
A Real-Time OS (RTOS) is defined by its timing guarantees — hard real-time systems (like pacemakers) must never miss a deadline, while soft real-time systems tolerate occasional misses.
Q4.What is the main difference between multiprogramming and multitasking?
Multiprogramming's goal is CPU utilization — never let the CPU sit idle when a job is waiting on I/O. Multitasking builds on this by rapidly time-slicing between programs so a user perceives them running at the same time.
Q5.A Time-Sharing Operating System achieves the illusion of multiple users running programs simultaneously by:
Time-sharing divides CPU time into small quantums and cycles through processes/users so fast that each one appears to have continuous, exclusive access to the CPU.
Q6.Which of these is NOT typically considered a core function of an Operating System?
Compiling code is the job of a compiler (application-level software), not the OS itself. The OS manages processes, memory, files, and devices — it doesn't translate source code.
Q7.In a Multiprocessing OS, what characteristic distinguishes it from a single-CPU multiprogramming system?
Multiprocessing specifically refers to systems with multiple CPUs working together (often sharing memory), which increases throughput and provides fault tolerance compared to a single-CPU multiprogrammed system.
Q8.A Distributed Operating System's key goal is to:
Distributed OS's defining feature is transparency — it coordinates multiple independent machines connected over a network so that, from the user's perspective, they behave like one unified system.

Process Management

A process is a program in execution — it includes the program code, current activity (program counter, registers), a stack, a data section, and a heap.

Process vs Program

A program is a passive entity: static code sitting on disk. A process is an active entity: the program loaded into memory, with its own execution context and resources. The same program can spawn multiple independent processes.

Process States

admit interrupt New -----------> Ready <-----------+ | ^ | dispatch| | | v | scheduler | Running | | I/O or event | | I/O or event wait v | completion Waiting -------------+ Running --exit--> Terminated
StateMeaning
NewProcess is being created
ReadyProcess is waiting to be assigned to the CPU (in the ready queue)
RunningInstructions are being executed by the CPU
Waiting (Blocked)Process is waiting for some event (I/O completion, signal) to occur
TerminatedProcess has finished execution

Process Control Block (PCB)

The OS represents each process with a PCB — a data structure holding all the information needed to manage the process:

  • Process ID (PID), process state
  • Program Counter (address of next instruction)
  • CPU registers, CPU scheduling info (priority)
  • Memory management info (page tables, base/limit registers)
  • Accounting info, I/O status info (open files, allocated devices)

Schedulers

SchedulerRoleFrequency
Long-term (Job Scheduler)Selects which processes are brought into the ready queue from disk (controls degree of multiprogramming)Runs infrequently (seconds/minutes)
Short-term (CPU Scheduler)Selects which ready process gets the CPU nextRuns very frequently (milliseconds)
Medium-term SchedulerSwaps processes in/out of memory to manage the degree of multiprogramming (swapping)Runs occasionally, as needed

Context Switching

A context switch is the act of saving the state (PCB) of the currently running process and loading the saved state of the next process to run. It is pure overhead — no useful work is done during the switch itself — so minimizing context switch time is a scheduling design goal.

Process Creation: fork() (Unix/Linux)

#include <stdio.h> #include <unistd.h> int main() { int pid = fork(); // creates a child process (a copy of the parent) if (pid == 0) { printf("Child process\n"); // pid==0 only in the CHILD } else if (pid > 0) { printf("Parent process, child PID = %d\n", pid); // pid = child's PID in PARENT } else { printf("fork failed\n"); } return 0; }

fork() returns twice: 0 in the child process, and the child’s PID in the parent process. If a program calls fork() n times, it creates 2n total processes (including the original).

Common gotcha: after fork(), both parent and child continue executing from the same point in the code — the line right after the fork call — not from the beginning of main().

10 questions
Q1.What is the key difference between a program and a process?
A program is static code sitting on disk. A process is that program loaded into memory and actively executing, with its own PC, registers, stack, and resources.
Q2.Which process state indicates that a process is waiting only for CPU availability (not for I/O)?
Ready means the process has everything it needs except the CPU — it's sitting in the ready queue waiting to be dispatched by the scheduler. Waiting/Blocked specifically means it's stalled on an event like I/O.
Q3.What does a Process Control Block (PCB) store?
The PCB is the OS's per-process bookkeeping structure — it holds everything needed to pause and later correctly resume a process: state, PC, registers, memory info, and I/O/accounting details.
Q4.Which scheduler controls the degree of multiprogramming by deciding which jobs are admitted into the ready queue?
The long-term (job) scheduler decides which jobs from the disk/job pool get loaded into memory as processes, directly controlling how many processes compete for the CPU (the degree of multiprogramming).
Q5.Why is a context switch considered pure overhead?
During a context switch, the CPU is busy saving the old process's PCB and loading the new one's — no actual user-facing computation happens in that window, so it's overhead the OS tries to minimize.
Q6.In the C code `int pid = fork();`, what does `pid == 0` indicate?
fork() returns 0 specifically in the child process. In the parent process, it returns the child's actual PID (a positive number). A negative return value indicates failure.
Q7.If a program calls fork() three times in sequence (with no conditional branching based on pid), how many total processes exist afterward?
Each fork() call doubles the number of processes, since every existing process (parent and all children so far) calls fork() again. After 3 calls: 2^3 = 8 total processes.
Q8.What is the role of the medium-term scheduler?
The medium-term scheduler temporarily removes (swaps out) processes from memory to reduce the multiprogramming load, and later swaps them back in — useful when memory is overcommitted.
Q9.After a fork() call in Unix, from which point does the child process begin execution?
Both parent and child resume execution right after the fork() call — they share the same code from that point onward, differentiated only by the return value of fork() (0 vs child PID).
Q10.Which of the following is TRUE about the short-term (CPU) scheduler?
The short-term scheduler operates at a very high frequency (often every few milliseconds) to pick the next process from the ready queue to actually run on the CPU — this is the core of CPU scheduling algorithms like Round Robin, SJF, etc.

CPU Scheduling

CPU Scheduling decides which process in the ready queue gets the CPU next, aiming to maximize utilization and throughput while minimizing waiting and response time.

Key Scheduling Metrics

MetricFormula
Completion Time (CT)Time at which a process finishes execution
Turnaround Time (TAT)CT − Arrival Time
Waiting Time (WT)TAT − Burst Time
Response Time (RT)Time of first CPU allocation − Arrival Time

1. First Come First Serve (FCFS)

Processes are executed strictly in order of arrival, non-preemptive. Simple but suffers from the convoy effect: a long process at the front makes all shorter processes behind it wait unnecessarily long.

Example: P1(burst=24), P2(burst=3), P3(burst=3), all arrive at t=0 Gantt Chart: | P1 (0-24) | P2 (24-27) | P3 (27-30) | WT(P1)=0, WT(P2)=24, WT(P3)=27 -> Average WT = 17 (Convoy effect: P2 and P3 are stuck waiting behind the long P1)

2. Shortest Job First (SJF)

Picks the process with the smallest burst time next. Non-preemptive by default. Provably gives the minimum average waiting time among non-preemptive algorithms — but requires knowing burst times in advance (often predicted via exponential averaging) and can cause starvation for long processes.

3. Shortest Remaining Time First (SRTF)

The preemptive version of SJF — if a new process arrives with a burst time shorter than the remaining time of the currently running process, the CPU is preempted immediately. Optimal for minimizing average waiting time, but incurs more context-switch overhead and can still starve long processes.

4. Round Robin (RR)

Each process gets a fixed time slice (time quantum); if it doesn’t finish, it goes to the back of the ready queue. Preemptive and fair — ideal for time-sharing systems.

Example: P1(burst=5), P2(burst=4), quantum=2 Gantt Chart: | P1(0-2) | P2(2-4) | P1(4-7) | P2(7-8) | Note: if quantum is TOO LARGE -> behaves like FCFS if quantum is TOO SMALL -> excessive context-switch overhead

5. Priority Scheduling

Each process is assigned a priority; the CPU is given to the highest-priority ready process. Can be preemptive or non-preemptive. Risk: starvation of low-priority processes — solved using aging (gradually increasing the priority of processes that wait too long).

Preemptive vs Non-Preemptive

Non-PreemptivePreemptive
DefinitionOnce CPU is given to a process, it keeps it until it finishes or blocks on I/OCPU can be forcibly taken away and given to another process
ExamplesFCFS, non-preemptive SJFSRTF, Round Robin, preemptive Priority
OverheadLower (fewer context switches)Higher (more context switches) but more responsive
Interview tip: If asked to compute average waiting/turnaround time, always draw the Gantt chart first — most mistakes come from misordering execution, not from the arithmetic.

SECTION QUIZ

10 questions
Q1.What is the formula for Turnaround Time (TAT)?
Turnaround Time measures the total time from when a process arrives to when it finishes, so it's Completion Time minus Arrival Time.
Q2.Which scheduling algorithm suffers most notably from the 'convoy effect'?
FCFS is non-preemptive and strictly order-based, so if a long process arrives first, all shorter processes behind it are forced to wait — this pile-up is the classic convoy effect.
Q3.Which algorithm provably gives the minimum average waiting time among non-preemptive scheduling algorithms?
SJF (non-preemptive) is proven optimal for minimizing average waiting time when compared among non-preemptive algorithms, precisely because it always tackles the shortest available job next.
Q4.What is the main risk of using Shortest Job First (SJF) or SRTF scheduling?
Because SJF/SRTF always favor the shortest job, a steady stream of short processes can indefinitely delay a long process — this is starvation.
Q5.In Round Robin scheduling, what happens if the time quantum is set far too large?
If the quantum is larger than any process's burst time, each process finishes within its first turn — meaning no process is ever preempted, which behaves identically to FCFS.
Q6.What technique is commonly used to prevent starvation in Priority Scheduling?
Aging solves starvation by incrementally boosting a waiting process's priority the longer it waits, eventually guaranteeing it gets scheduled.
Q7.Which of the following is a preemptive scheduling algorithm?
Round Robin is inherently preemptive — each process is interrupted when its time quantum expires, even if it hasn't finished, and placed back in the ready queue.
Q8.Given P1(burst=24) and P2(burst=3) both arriving at t=0, executed via FCFS in order P1 then P2, what is P2's waiting time?
In FCFS with P1 first, P1 runs from 0-24. P2 must wait the entire time P1 runs before it even starts, so its waiting time equals P1's burst time: 24.
Q9.What is Response Time in CPU scheduling?
Response Time specifically measures how long a process waits before it is FIRST given the CPU — important for interactive/time-sharing systems where quick initial response matters, distinct from total turnaround.
Q10.Why does SRTF generally have more overhead compared to non-preemptive SJF?
Because SRTF re-evaluates the shortest remaining time on every new arrival and can preempt the currently running process, it triggers many more context switches than non-preemptive SJF, which never interrupts a running process once started.

Process Synchronization

Process Synchronization deals with coordinating concurrent processes/threads that share resources, to avoid inconsistent results.

Race Condition

A race condition occurs when multiple processes/threads access and modify shared data concurrently, and the final result depends on the unpredictable timing/order of execution.

Example: two threads both execute "counter++" on a shared variable counter=5 Thread A reads counter (5) Thread B reads counter (5) <- both read the SAME stale value Thread A writes counter+1 (6) Thread B writes counter+1 (6) <- one increment is LOST! Expected 7, got 6

Critical Section Problem

The critical section is the part of code where shared resources are accessed. A correct solution must satisfy:

  • Mutual Exclusion — no two processes can be in their critical section at the same time.
  • Progress — if no process is in the critical section, one of the processes wishing to enter must be able to, without indefinite postponement.
  • Bounded Waiting — there must be a limit on how many times other processes can enter their critical section before a waiting process gets its turn.

Semaphores

A semaphore is an integer variable accessed only through two atomic operations: wait() (P, decrements) and signal() (V, increments).

wait(S) { signal(S) { while (S <= 0) S++; ; // busy wait } S--; }
TypeDescription
Binary Semaphore (Mutex)Value restricted to 0 or 1 — used for mutual exclusion (lock/unlock)
Counting SemaphoreValue can range over an unrestricted domain — used to control access to a resource with multiple instances (e.g., a pool of N identical printers)

Mutex vs Semaphore (classic interview question)

MutexSemaphore
PurposeLocking mechanism, mutual exclusion onlySignaling mechanism, can allow N processes access
OwnershipOnly the thread that locked it can unlock itAny process/thread can signal, regardless of who waited
Value rangeBinary (locked/unlocked)Can be binary or counting (integer)

Classic Synchronization Problems

  • Producer-Consumer Problem — producers add items to a shared bounded buffer, consumers remove them; must avoid producing into a full buffer or consuming from an empty one. Solved using two counting semaphores (empty, full) plus a mutex.
  • Reader-Writer Problem — multiple readers can read shared data simultaneously, but a writer needs exclusive access (no readers or other writers).
  • Dining Philosophers Problem — models resource contention/deadlock: N philosophers alternate between thinking and eating, needing two shared forks to eat, illustrating deadlock and starvation risks in concurrent resource allocation.

Peterson’s Solution

A classic software-only solution to the critical section problem for two processes, using a turn variable and a flag[] array to enforce mutual exclusion, progress, and bounded waiting without hardware support. (Note: not reliable on modern hardware with instruction reordering / compiler optimizations without additional memory barriers.)

SECTION QUIZ

10 questions
Q1.What is a race condition?
A race condition specifically arises when the final result of concurrent operations on shared data depends on the order/timing of execution — classic symptom: lost updates, like two threads incrementing a shared counter and one increment getting overwritten.
Q2.Which of these is NOT one of the three required conditions for a correct critical section solution?
The three required properties are Mutual Exclusion, Progress, and Bounded Waiting. Maximum throughput is a performance goal, not a correctness requirement of the critical section problem.
Q3.What does the wait() (P) operation do to a semaphore?
wait() (also called P, from Dutch 'proberen') decrements the semaphore. If the value would go below zero (in blocking implementations) or is already zero, the calling process waits/blocks.
Q4.What is the key difference between a binary semaphore and a mutex?
While both restrict access to 0/1 in the binary case, a mutex enforces ownership (only the thread that acquired the lock can release it), whereas a semaphore can be signaled by any thread/process, regardless of who performed the wait.
Q5.In the Producer-Consumer problem using semaphores, what is the purpose of the 'empty' and 'full' counting semaphores?
The 'empty' semaphore tracks available empty slots (producer waits on this before adding), and 'full' tracks filled slots (consumer waits on this before removing) — together with a mutex they prevent overproduction into a full buffer or consuming from an empty one.
Q6.What is the central challenge illustrated by the Dining Philosophers problem?
Dining Philosophers models N processes each needing two shared resources (forks) to proceed — if everyone grabs their left fork simultaneously, all wait forever for their right fork, illustrating deadlock and starvation risks in concurrent systems.
Q7.In the Reader-Writer problem, which of the following access patterns is allowed?
Since reads don't modify data, multiple readers can safely access shared data concurrently. But a writer requires fully exclusive access — no readers or other writers can be active during a write.
Q8.What does Peterson's Solution use to achieve mutual exclusion for two processes?
Peterson's Solution is a purely software-based algorithm using a shared 'turn' variable (whose turn it is) and a 'flag' array (who wants to enter), without needing any special hardware instructions — though it's not fully reliable on modern hardware without memory barriers.
Q9.A counting semaphore would be most appropriate for managing access to:
Counting semaphores generalize beyond binary lock/unlock — they're ideal when there are multiple (N) interchangeable instances of a resource, allowing up to N processes concurrent access before others must wait.
Q10.What lost-update scenario best illustrates why a race condition is problematic?
The classic race condition example: if two threads read the same stale value of a shared counter before either writes back their incremented result, one thread's update effectively gets silently lost.

Deadlocks

A deadlock is a situation where a set of processes are blocked, each holding a resource and waiting to acquire a resource held by another process in the set — forming a cycle with no process able to proceed.

Four Necessary Conditions (Coffman Conditions)

All four must hold simultaneously for a deadlock to occur:

ConditionMeaning
Mutual ExclusionAt least one resource must be held in a non-shareable mode
Hold and WaitA process holding at least one resource is waiting to acquire additional resources held by others
No PreemptionResources cannot be forcibly taken away from a process; only voluntarily released
Circular WaitA set of processes {P0, P1, ..., Pn} exists such that P0 waits for a resource held by P1, P1 waits for P2, ..., Pn waits for P0

Resource Allocation Graph (RAG)

Notation: P1 -> R1 means "P1 is requesting R1" R1 -> P1 means "R1 is allocated to P1" Example showing a deadlock (cycle): P1 ---request---> R1 ---allocated---> P2 ^ | | request | v R2 <---allocated--- (assigned to P1) R2 If a cycle exists in a RAG WITH single-instance resources -> deadlock is guaranteed. If resources have multiple instances, a cycle is necessary but NOT sufficient for deadlock.

Deadlock Handling Strategies

StrategyApproach
PreventionDesign the system so at least one of the four Coffman conditions can never hold (e.g., request all resources at once to break Hold-and-Wait)
AvoidanceGrant resource requests only if the resulting state is "safe" (won't lead to deadlock) — uses algorithms like Banker's Algorithm
Detection & RecoveryAllow deadlocks to occur, periodically check for cycles/deadlock via algorithms, then recover (kill a process or preempt a resource)
Ignorance (Ostrich Algorithm)Assume deadlocks are rare enough to ignore — used by many general-purpose OSes like Unix/Windows for simplicity

Banker’s Algorithm

An avoidance algorithm that simulates resource allocation before actually granting it, checking whether the system remains in a safe state (there exists some order in which all processes can finish without deadlock). Requires the OS to know the maximum resource demand of each process in advance.

Core idea: 1. Process requests resources. 2. OS pretends to grant the request. 3. Check if a SAFE SEQUENCE still exists (i.e. some order lets every process finish using currently available + eventually released resources). 4. If yes -> actually grant the request. If no -> process must wait (request is rolled back).

Safe State vs Unsafe State

A safe state is one where the system can allocate resources to each process (up to its max demand) in some order and avoid deadlock. An unsafe state does not necessarily mean deadlock has occurred — but it means there is a possibility of deadlock, since no safe sequence can be guaranteed.

Common confusion: Unsafe state ≠ deadlock. All deadlocked states are unsafe, but not all unsafe states lead to deadlock — the OS is just being conservative by refusing to enter an unsafe state.

SECTION QUIZ

10 questions
Q1.Which of the following is NOT one of the four Coffman conditions necessary for deadlock?
The actual condition is 'No Preemption' (resources cannot be forcibly taken away) — the presence of preemption capability would actually help PREVENT deadlock, not cause it.
Q2.For deadlock to occur, how many of the four Coffman conditions must hold simultaneously?
All four conditions (Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait) must hold simultaneously for deadlock to be possible. Breaking even one of them prevents deadlock.
Q3.In a Resource Allocation Graph with only single-instance resource types, what does a cycle indicate?
With single-instance resources, a cycle in the RAG is both necessary AND sufficient for deadlock — its presence guarantees a deadlock. This is not true when resources have multiple instances, where a cycle is necessary but not sufficient.
Q4.What is the main idea behind the 'Ostrich Algorithm' approach to deadlocks?
The Ostrich Algorithm 'buries its head in the sand' — it assumes deadlocks occur infrequently enough that it's not worth the overhead of prevention/avoidance/detection. Many general-purpose systems like Unix and Windows take this pragmatic approach.
Q5.What must the OS know in advance to run the Banker's Algorithm?
Banker's Algorithm requires each process to declare its maximum possible resource need upfront, so the OS can simulate whether granting a request keeps the system in a safe state.
Q6.What does it mean for a system to be in a 'safe state'?
A safe state means there's a guaranteed sequence in which every process can eventually get all the resources it needs (up to its max) and finish, without ever hitting deadlock — even if not all resources are free right now.
Q7.Is an 'unsafe state' the same as a deadlocked state?
This is a classic point of confusion: an unsafe state means there's no guaranteed safe sequence, so deadlock COULD happen — but it doesn't mean deadlock has definitely occurred. All deadlocks are unsafe states, but not all unsafe states become deadlocks.
Q8.Which deadlock handling strategy involves periodically checking the system for cycles and then killing a process or preempting a resource?
Detection and Recovery lets deadlocks happen, then periodically runs a detection algorithm (like cycle detection in the RAG) and recovers by killing a process or forcibly preempting a resource from one.
Q9.Which strategy breaks the 'Hold and Wait' condition by requiring a process to request all its needed resources at once, before execution begins?
Prevention strategies target breaking one of the four necessary conditions structurally. Requiring all resources to be requested upfront (instead of incrementally) directly eliminates the possibility of 'hold and wait'.
Q10.In a Resource Allocation Graph, what does an edge from R1 to P2 represent?
By RAG convention, an edge FROM a resource TO a process (R1 -> P2) represents an assignment edge — meaning R1 is currently allocated to P2. An edge from a process to a resource (P -> R) represents a request edge instead.

Memory Management

Memory Management is the OS function responsible for allocating and deallocating memory space to processes, ensuring efficient use of RAM while keeping processes isolated from each other.

Contiguous Memory Allocation

Early systems allocated each process a single contiguous block of memory.

SchemeDescription
Fixed (Static) PartitioningMemory divided into fixed-size partitions at boot time; each partition holds one process — simple but causes internal fragmentation (wasted space inside a partition when the process is smaller than the partition)
Dynamic PartitioningPartitions are created dynamically, sized exactly to each process's needs — avoids internal fragmentation but causes external fragmentation (free memory scattered in small chunks, none large enough for a new request)

Fragmentation

  • Internal Fragmentation — wasted space within an allocated block (block is bigger than what's actually needed).
  • External Fragmentation — wasted space between allocated blocks; total free memory may be sufficient but it's not contiguous.

Allocation Strategies (Dynamic Partitioning)

StrategyApproach
First FitAllocate the first free block that is large enough
Best FitAllocate the smallest free block that is large enough (minimizes leftover space, but that leftover is often too small to be useful, worsening external fragmentation over time)
Worst FitAllocate the largest free block available (leaves the biggest possible leftover, hoping it stays useful)

Paging

Paging solves external fragmentation by dividing physical memory into fixed-size blocks called frames, and logical memory (a process’s address space) into blocks of the same size called pages. A page table maps each page to its corresponding frame.

Logical Address = (Page Number, Offset) CPU generates logical address | v Page Number ---> [Page Table] ---> Frame Number | v Physical Address = (Frame Number, Offset) Example: Page size = 4KB, logical address 0x2050 Page Number = 0x2050 / 4096 = 2 Offset = 0x2050 % 4096 = 80 Look up page 2 in the page table -> get Frame number, say Frame 5 Physical Address = Frame 5, Offset 80

Paging eliminates external fragmentation entirely (since any free frame can hold any page), but introduces a small amount of internal fragmentation in the last page of a process (if it doesn’t exactly fill a page).

Segmentation

Segmentation divides a process into logical segments (code, stack, heap, data) of variable size, matching how programmers naturally think about a program's structure. Each segment has a base and limit register. Unlike paging, segmentation can suffer from external fragmentation since segments vary in size.

PagingSegmentation
Division basisFixed-size physical blocks (frames/pages)Logical program divisions (variable size)
FragmentationInternal onlyExternal only
Programmer visibilityInvisible to the programmerReflects logical program structure, more visible
Interview tip: A very common question is "Why does paging avoid external fragmentation but segmentation doesn't?" — because paging uses fixed, uniform block sizes (any free frame fits any page), whereas segmentation uses variable sizes (a free gap may not exactly fit the next segment needed).

SECTION QUIZ

10 questions
Q1.What is internal fragmentation?
Internal fragmentation happens when an allocated block of memory is larger than required, wasting the unused space inside that block — for example, in fixed-size partitioning or paging's last page.
Q2.Which memory allocation strategy allocates the smallest free block that is still large enough for the request?
Best Fit specifically searches for the smallest suitable free block to minimize leftover space — though ironically this can create many tiny unusable fragments over time.
Q3.How does paging eliminate external fragmentation?
Because paging uses uniform, fixed-size units (frames and pages), any free frame can accommodate any page — there's no need for a contiguous chunk of a specific size, which is what causes external fragmentation in variable-sized schemes.
Q4.What does a page table do?
The page table is the core data structure of paging — for each page number in a process's logical address space, it holds the corresponding physical frame number where that page currently resides in memory.
Q5.Given a page size of 4KB and logical address 0x2050, what is the page number?
Page number = logical address / page size = 0x2050 (8272 in decimal) / 4096 = 2 (integer division), with a remainder used as the offset.
Q6.What is the key difference between paging and segmentation in terms of fragmentation?
Paging's fixed block sizes mean any leftover waste is confined within the last page (internal). Segmentation's variable-sized segments can leave unusable gaps between segments in memory (external), similar to dynamic partitioning.
Q7.Why does segmentation better reflect a programmer's view of a program compared to paging?
Segmentation divides memory according to a program's logical structure (functions/code segment, stack, heap, global data) rather than arbitrary fixed-size chunks, which maps more naturally to how a programmer conceptually organizes a program.
Q8.In dynamic (variable) partitioning, why does Best Fit not always perform better than First Fit in practice?
While Best Fit minimizes leftover space per allocation, it tends to create many very small leftover fragments across memory that are too small to satisfy future requests — paradoxically increasing external fragmentation despite its name.
Q9.What information does each segment typically have associated with it in segmentation?
Each segment has a base (starting physical address) and a limit (length of the segment), which together are used to translate a logical (segment, offset) address into a physical address and to check bounds.
Q10.Which fragmentation type does the LAST page of a process typically introduce, if the process size isn't an exact multiple of the page size?
If a process's total size doesn't perfectly divide into whole pages, the final page is only partially used, wasting the remainder of that page — this is internal fragmentation, the only type paging is subject to.

Paging & Virtual Memory

Virtual Memory is a technique that gives processes the illusion of a large, contiguous address space, even if physical memory (RAM) is smaller or fragmented — by keeping only the actively-used portions of a process in RAM and the rest on disk.

Demand Paging

Pages are loaded into memory only when needed (referenced), not all at once when a process starts. Each page table entry has a valid/invalid bit:

  • Valid — the page is currently in physical memory.
  • Invalid — the page is not in memory (either not yet used, or swapped out to disk).

Page Fault

A page fault occurs when a process references a page marked invalid in the page table (not currently in RAM). The OS handles it by:

1. Trap to OS (page fault interrupt) 2. Check if the reference was valid (legal) or invalid (illegal access -> terminate process) 3. Find a free frame in physical memory (or select a victim page to evict, if memory is full) 4. Bring the required page in from disk into that frame 5. Update the page table (mark as valid, set frame number) 6. Restart the instruction that caused the page fault

Page Replacement Algorithms

When memory is full and a new page must be brought in, a page replacement algorithm decides which existing page to evict (the "victim").

AlgorithmStrategy
FIFO (First In First Out)Evicts the page that has been in memory the longest, regardless of usage
Optimal (OPT / MIN)Evicts the page that will not be used for the longest time in the future — theoretical best case, requires future knowledge (used as a benchmark)
LRU (Least Recently Used)Evicts the page that hasn't been used for the longest time in the past — approximates Optimal using historical access patterns

Belady’s Anomaly

A counter-intuitive phenomenon (specific to FIFO) where increasing the number of physical memory frames can increase the number of page faults, instead of decreasing them. LRU and Optimal do not suffer from Belady's Anomaly.

Reference string: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 With 3 frames (FIFO): 9 page faults With 4 frames (FIFO): 10 page faults <- MORE frames, MORE faults! (the anomaly)

Thrashing

Thrashing occurs when a process spends more time paging (swapping pages in/out) than executing actual instructions — typically caused by too many processes competing for too little physical memory, each not having enough frames for its working set (the set of pages it actively needs at a given time). Detected via a sharp drop in CPU utilization despite high degree of multiprogramming.

Working Set Model

The working set of a process is the set of pages it has referenced in the most recent time window Δ. The OS uses this to decide how many frames to allocate to a process — if the working set doesn't fit in the frames allocated, thrashing is likely.

Common confusion: a page fault is NOT necessarily an error condition — it's a completely normal part of demand paging. A page fault only becomes a genuine problem if it happens excessively, leading to thrashing.

SECTION QUIZ

10 questions
Q1.What is demand paging?
Demand paging is a lazy-loading strategy: instead of loading an entire process into memory upfront, pages are brought in from disk only when the CPU actually references them.
Q2.What does the valid/invalid bit in a page table entry indicate?
The valid/invalid bit tells the MMU whether a given page is currently loaded into a physical frame (valid) or not present in memory at all, requiring a page fault to bring it in (invalid).
Q3.What triggers a page fault?
A page fault is a hardware trap that occurs when a memory reference points to a page marked invalid in the page table — the OS must then step in to bring that page into memory.
Q4.Which page replacement algorithm requires knowledge of future memory references and serves mainly as a theoretical benchmark?
The Optimal algorithm evicts the page that won't be used for the longest time in the future — but since no real system can know the future, it's used purely as a theoretical best-case comparison point for other algorithms.
Q5.What is Belady's Anomaly?
Belady's Anomaly is the counter-intuitive result where adding MORE physical frames leads to MORE page faults under certain reference strings with FIFO replacement — it doesn't occur with LRU or Optimal.
Q6.What does LRU (Least Recently Used) page replacement evict?
LRU looks backward in time and evicts whichever page has gone the longest without being referenced, approximating the Optimal algorithm's forward-looking strategy using historical data instead.
Q7.What is thrashing?
Thrashing happens when there's so much page-fault activity (due to too many processes competing for too few frames) that the system spends nearly all its time swapping pages in and out rather than doing useful work — CPU utilization drops sharply despite high multiprogramming.
Q8.What is the 'working set' of a process?
The working set is a snapshot of which pages a process is ACTIVELY using in the recent past (within a window Δ) — it's used to estimate how many frames a process needs to avoid excessive page faulting.
Q9.Which of these page replacement algorithms does NOT suffer from Belady's Anomaly?
Belady's Anomaly is a known weakness specifically associated with FIFO. LRU and Optimal belong to a class of algorithms (stack algorithms) that are mathematically guaranteed not to exhibit this anomaly.
Q10.After a page fault is handled and the missing page is loaded into memory, what happens next?
Once the required page is brought into a free (or newly freed) frame and the page table is updated, the OS restarts the exact instruction that triggered the fault — from the CPU's perspective, execution just continues normally as if the page had always been there.

Disk Scheduling

Disk Scheduling algorithms decide the order in which pending disk I/O requests (each targeting a specific cylinder/track) are serviced, aiming to minimize total head movement (seek time) and/or average waiting time.

Why It Matters

A disk's read/write head must physically move across cylinders to reach each requested track. This mechanical movement (seek time) is the slowest part of a disk I/O operation, so scheduling requests intelligently has a large effect on overall performance.

1. FCFS (First Come First Serve)

Services requests strictly in the order they arrive in the queue. Simple and fair, but can result in a lot of unnecessary head movement (as seen in the diagram above, zig-zagging back and forth).

2. SSTF (Shortest Seek Time First)

Always services the request closest to the current head position next. Greatly reduces total head movement compared to FCFS, but can cause starvation of requests far from the current head position if closer requests keep arriving.

3. SCAN (Elevator Algorithm)

The head moves in one direction (say, toward the highest cylinder), servicing all requests along the way, until it reaches the end of the disk, then reverses direction and services requests on the way back — like an elevator.

Example: requests = 82, 170, 43, 140, 24, 16, 190 Head starts at 50, moving toward higher cylinders (direction = up), disk range 0-199 Order serviced: 82, 140, 170, 190, 199(end), 43, 24, 16 Total head movement = (199-50) + (199-16) = 149 + 183 = 332

4. C-SCAN (Circular SCAN)

Like SCAN, but instead of reversing direction at the end, the head jumps immediately back to the beginning (cylinder 0) and continues scanning in the same direction. This gives more uniform wait times across all cylinders (SCAN tends to favor cylinders near the middle, since they get visited on both sweeps).

Example: same requests, head starts at 50, moving up, disk range 0-199 Order serviced: 82, 140, 170, 190, 199(end), jump to 0, then 16, 24, 43 Total head movement = (199-50) + 199 + 43 = 149 + 199 + 43 = 391 (the jump from 199 to 0 is usually NOT counted as head movement, since it's a fast return, not a seek)

5. LOOK and C-LOOK

Optimizations of SCAN and C-SCAN respectively: instead of going all the way to the disk's physical end (cylinder 0 or max), the head only goes as far as the last request in that direction, then reverses (LOOK) or jumps back (C-LOOK). This avoids wasting movement on empty regions with no pending requests.

SECTION QUIZ

10 questions
Q1.What is the main goal of disk scheduling algorithms?
Since seek time (physical head movement) is the slowest part of disk I/O, scheduling algorithms aim to service pending requests in an order that minimizes total head movement and/or keeps waiting times fair.
Q2.Which disk scheduling algorithm services requests strictly in the order they arrive, regardless of their cylinder position?
FCFS (First Come First Serve) is the simplest approach — it ignores cylinder positions entirely and just processes requests in arrival order, which can cause a lot of unnecessary back-and-forth head movement.
Q3.What is the main risk of using SSTF (Shortest Seek Time First)?
Because SSTF greedily always jumps to the nearest request, a steady stream of nearby requests can indefinitely delay a distant request — this is starvation, similar to what happens with SJF in CPU scheduling.
Q4.In the SCAN algorithm, what happens when the head reaches the end of the disk?
SCAN moves in one direction servicing all requests until it hits the physical end of the disk, then reverses and sweeps back the other way — this back-and-forth pattern is why it's nicknamed the 'elevator algorithm'.
Q5.How does C-SCAN differ from standard SCAN?
C-SCAN (Circular SCAN) always sweeps in one consistent direction — after reaching the end, it jumps back to the start (cylinder 0) rather than reversing, which gives more uniform wait times across all cylinders compared to SCAN.
Q6.Why might C-SCAN provide more uniform waiting times than SCAN?
In standard SCAN, middle cylinders get serviced twice as often (once on each sweep direction) compared to cylinders near the edges, causing uneven waiting times. C-SCAN's one-directional sweep pattern spreads out waiting more evenly.
Q7.What is the key difference between SCAN and LOOK?
LOOK is an optimization of SCAN — instead of blindly continuing to the disk's absolute end/beginning, it 'looks' ahead and reverses direction as soon as there are no more pending requests in the current direction, avoiding wasted movement.
Q8.Given requests at cylinders 82, 170, 43 with the head starting at 50, which request would SSTF service FIRST?
SSTF always picks whichever pending request has the smallest seek distance from the current head position. |50-43|=7, |50-82|=32, |50-170|=120 — so 43 is closest and serviced first.
Q9.In C-LOOK, when the head reaches the last request in its current direction, what happens next?
C-LOOK combines C-SCAN's one-directional circular sweep with LOOK's efficiency — after servicing the last request in one direction, it jumps straight to the lowest pending request (not all the way to cylinder 0) and continues scanning in the same direction.
Q10.Why is FCFS generally considered inefficient for disk scheduling despite being simple and fair?
Because FCFS ignores cylinder proximity entirely, a poorly-ordered arrival sequence can send the head zig-zagging across the disk (e.g., servicing cylinder 170, then 43, then 140), racking up far more total movement than a proximity-aware algorithm like SSTF or SCAN would.

File Systems

A file system defines how data is stored, organized, named, and retrieved on a storage device. It manages files, directories, free space, and metadata (permissions, timestamps, size).

File Allocation Methods

Determines how a file's data blocks are physically arranged on disk.

MethodDescriptionDrawback
Contiguous AllocationEach file occupies a single contiguous range of blocks on diskFast sequential access, but suffers external fragmentation; file growth is hard to accommodate
Linked AllocationEach block contains a pointer to the next block in the file (like a linked list)No external fragmentation, but slow random access (must traverse from the start) and pointer overhead per block
Indexed AllocationAn index block stores pointers to all of a file's data blocksSupports fast direct access; extra space needed for the index block itself

Inodes and Indexed Allocation (Unix-style)

An inode (index node) stores a file’s metadata (size, permissions, timestamps) plus pointers to its data blocks. To support both small and very large files efficiently, inodes typically use a mix of pointer types:

  • Direct pointers — point straight to a data block. Fast, but limited in count (a fixed number of direct pointers per inode).
  • Single indirect pointer — points to a block that itself contains more pointers to data blocks, extending the file size an inode can address.
  • Double/triple indirect pointers — point to blocks of pointers to blocks of pointers (and so on), allowing enormous files to be addressed while keeping the inode itself small and fixed-size.

Directory Structures

StructureDescription
Single-LevelOne directory for all files — simple but no organization, no duplicate filenames allowed
Two-LevelEach user gets their own directory — solves naming collisions between users, but no sub-organization within a user's space
Tree-StructuredDirectories can contain subdirectories, forming a hierarchy (like modern file systems: /home/user/docs/)
Acyclic GraphAllows shared subdirectories/files via links, while still disallowing cycles

Hard Links vs Soft (Symbolic) Links

Free Space Management

  • Bitmap (Bit Vector) — one bit per block: 0 = free, 1 = allocated. Compact and fast to scan for contiguous free regions.
  • Linked List — free blocks are chained together in a list; no extra space needed elsewhere, but traversal to find free blocks is slower.
  • Grouping — the first free block stores addresses of several other free blocks, reducing traversal overhead.
Common interview confusion: a hard link is NOT a "copy" of a file — both the original name and the hard link point to the exact same inode/data on disk. Deleting one name just decrements the inode's link count; the data is only actually freed when the count reaches zero.

SECTION QUIZ

10 questions
Q1.What is the main drawback of Contiguous File Allocation?
Since contiguous allocation requires a single unbroken range of blocks, free space can become fragmented over time (external fragmentation), and growing a file in place is often impossible if adjacent blocks are already used.
Q2.In Linked Allocation, how is a file's data physically organized on disk?
Linked allocation avoids external fragmentation by chaining blocks together like a linked list — each block holds a pointer to the next block belonging to that file, wherever it may be on disk.
Q3.What is a key drawback of Linked Allocation compared to Indexed Allocation?
Because each block only knows the location of the NEXT block, accessing, say, the 100th block of a file requires traversing all 99 blocks before it — this makes random access very slow compared to indexed allocation's direct lookups.
Q4.What does an inode store?
An inode is the core metadata structure for a file in Unix-style filesystems — it holds attributes like size and permissions, along with direct/indirect pointers that locate the actual data blocks.
Q5.Why do inodes use indirect pointers (single, double, triple) in addition to direct pointers?
A fixed, small number of direct pointers alone couldn't address very large files. Indirect pointers let an inode reach many more data blocks (via blocks-of-pointers) without growing the inode's own fixed size — striking a balance between small-file speed and large-file capacity.
Q6.What is the fundamental difference between a hard link and a soft (symbolic) link?
A hard link is essentially another name for the exact same inode/data — no distinction from the 'original.' A soft link is a separate, small file that just stores a path string pointing to the target — if that path becomes invalid, the symlink breaks.
Q7.What happens to a hard link if the original file is deleted?
Inodes track a reference/link count. Deleting one name (including the 'original') just decrements that count — the actual data blocks are only freed once the count reaches zero, meaning any remaining hard links still fully work.
Q8.Which directory structure allows directories to contain subdirectories, forming a hierarchy like /home/user/docs/?
A tree-structured directory is what nearly all modern filesystems use — directories can nest arbitrarily deep, forming a hierarchical path structure like /home/user/docs/file.txt.
Q9.In free space management, what does a Bitmap (bit vector) approach use to track free blocks?
A bitmap dedicates exactly one bit per disk block, making it very compact, and it's fast to scan for contiguous runs of free blocks — a common real-world technique for free space tracking.
Q10.Can a soft (symbolic) link point to a file on a different filesystem or partition than the one it resides on?
Since a symlink is just a small file containing a path (not a direct inode reference), it can point anywhere the path resolves to — including files on entirely different filesystems or partitions, unlike hard links which are inode-based and filesystem-bound.
